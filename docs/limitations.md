# Limitations

- PolicyTwin is a software verification aid, not legal advice. Human approval is required before real policy deployment.
- Only the synthetic English SaaS refund policy and bundled trusted TypeScript fixture are supported.
- The current repository has not performed a fresh GPT-5.6 request or live Codex SDK repair. The pinned SDK-compatible adapter, signed RPC client, real TLS 1.3 mutual-authentication transport, bounded supervisor, and durable SQLite replay store are covered offline with ephemeral certificates, generated test signing keys, a signed `FAIL` executor double, and trusted fixture runners; none is presented as live Codex evidence. Separate worker/verifier/egress Dockerfiles, a candidate-to-inspected-ID shell-free Docker driver, a fixed two-file-overlay/no-network-verifier plan, a validate-only worker entrypoint, command-backed run-capability authentication, and a Responses-only reverse broker are statically or locally tested. The driver is connected only to the prepared static lifecycle and a stateful fake daemon, not to a signed live RPC result. The supervisor seals the worker image and request maxima; memory equals memory+swap, regular-file writes and the single local log file share bounded ceilings, and one prepare/worker/verifier execution deadline is reflected in the plan. All prepared process plans explicitly use `restart=no`; the fake-daemon driver requires zero restarts and a stable ID/PID/start timestamp, with egress reobserved around worker execution and before stop. A required CPU-controller port and fake-only ledger aggregate post-baseline egress, worker, and verifier values using `bigint`, bind a static proof to the request, Docker binding, and three role identities, and delay receipt validation until proof finalization. The proof explicitly reports no cumulative enforcement, hard limit, overshoot bound, or containment, and cannot pass the live gate. No real cgroup path, `cpu.stat`, baseline timing, polling latency, freeze/kill permission, or actuation has been observed. Cleanup has a separate bounded grace period, and controller-cleanup failure blocks the lifecycle. No immutable image has run; real restart/swap/`fsize`/log-limit behavior, the broker's internal TLS and upstream identity, process/cgroup limits, or teardown are not dynamically proved. The web process has no live repair route and the host live-backend factory always rejects. The repository also executes checksum-pinned OPA 1.18.2 locally over the 41-case accepted corpus.
- Worker RPC v2 CPU evidence schema v2 is an offline contract, not live CPU evidence. It strictly validates a request-bound global `CLOCK_MONOTONIC_RAW_NS` transcript, egress/worker overlap, verifier ordering, linked samples, success arithmetic, and closed pre-execution/controller/overage/containment failure outcomes. Synthetic unit evidence uses test signing keys; the loopback mTLS v2 path intentionally signs only typed pre-execution `FAIL`, and the generic supervisor rejects PASS. No artifact proves that Linux supplied the timestamps or samples, a controller polled or contained processes, Docker executed, cgroups were released, or cumulative CPU was enforced. Contract-implemented flags are true while observation/dynamic flags remain false. `hardLimitEnforced` and `overshootBounded` remain false, and `verify:live` remains blocked.
- The v2 transport capability closes public string/copy/wrapper admission and later caller-option mutation only. Its actual TLS factory, validated scalar snapshot, defensively copied in-memory TLS material, and private `WeakSet` are co-located and no arbitrary registrar exists, but this is not a sandbox against arbitrary code already executing inside the trusted host process. Actual security still depends on TLS 1.3 peer validation and pins, the dedicated Ed25519 trust bundle, durable replay, and the disabled PASS/live gate.
- The zero-drift fixed fixture is evaluation-only. It is not post-Codex evidence.
- The current mutation score uses the reference evaluator, not OPA.
- The evidence package is intentionally `PARTIAL_OFFLINE` and `FAIL`.
- SQLite persistence, anonymous browser-session isolation, versioned Decision Queue/source writes, the six-screen web workspace, production build, and local Chrome navigation/keyboard/responsive checks exist. Anonymous projects have a process-local 24-hour TTL and 128-project cap, but there is no authenticated identity or distributed quota. The 14-to-30 text edit persists only as a blocked v5 `DRAFT`; its impact is a reference preview, not accepted PolicyIR, fresh OPA evidence, or a repair result. A full accessibility audit, multi-user authentication, container health, deployment, public URLs, video, and challenge submission remain incomplete.
- The deterministic 38-file USTAR download and 14-to-30 preview belong only to the seeded reference ambiguity choices. The archive is uncompressed and generated in memory from validated text evidence; it deliberately excludes screenshots, SQLite data, transient logs, and arbitrary files. A semantic PolicyIR fingerprint exposes a mismatch and blocks the draft when a browser accepts different valid choices; session-specific proof generation is not implemented yet.
- SHA-256 evidence hashes provide integrity, not origin authentication. `LIVE_VERIFIED` additionally requires a trusted Ed25519 attestation whose private key must stay outside the repository and whose default freshness window is 24 hours; the live signer is not implemented yet.
- The standalone web, worker, verifier, and egress Dockerfiles and daemon-free static contract exist, but the immutable Node base is unset and the local Docker Desktop Linux daemon is unavailable. `container:verify` covers the web image and SQLite restart path. `worker:verify` is the non-live worker/verifier gate with a fresh request-bound internal network, inspected-ID cleanup, required Linux cgroup v2 process-tree evidence, and role-local post-exit CPU comparisons that are not aggregate enforcement. `egress:verify` is a separate non-live proxy gate with fresh internal/outbound networks, private ephemeral secrets, the same cgroup/absence proof, and a TLS 1.3 certificate probe that writes no HTTP. Proxy outbound traffic is not measured, so upstream absence is not proved. All three dynamic reports currently fail before Docker. The selected Docker/BuildKit identity, exact image environment/inspect compatibility, signed real-Linux three-role cumulative CPU proof, actual egress runtime, forced termination, real Codex headers, and OpenAI upstream remain unverified.
- No project license has been selected because owner acceptance is required.
- Arbitrary repository upload/execution, multiple policy domains, multiple languages, and production auto-deployment are excluded from the MVP.
