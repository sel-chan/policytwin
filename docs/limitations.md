# Limitations

- PolicyTwin is a software verification aid, not legal advice. Human approval is required before real policy deployment.
- Only the synthetic English SaaS refund policy and bundled trusted TypeScript fixture are supported.
- The PolicyIR structure is now single-sourced in Zod for runtime admission, deterministic JSON Schema generation, and the OpenAI Structured Outputs model projection. This removes three independently maintained structural shapes, but it does not make model output semantically correct: source traceability, field/value compatibility, IDs and references, patch targets, exact input schema, ambiguity state, and golden-case agreement are still checked deterministically. Provider acceptance also remains unverified until a fresh GPT-5.6 run succeeds.
- The Responses adapter distinguishes explicit refusal, incomplete generation (`max_output_tokens` or `content_filter`), upstream error, and non-completed status from recoverable structured-output defects and does not retry those terminal outcomes. It also cross-checks returned message text with the SDK `output_text` aggregate when output items are present. These are fixture-backed local contract checks; no actual GPT-5.6 terminal response has been observed in this repository.
- The current repository has not performed a fresh GPT-5.6 request or live Codex SDK repair. The pinned SDK-compatible adapter, signed RPC client, real TLS 1.3 mutual-authentication transport, bounded supervisor, and durable SQLite replay store are covered offline with ephemeral certificates, generated test signing keys, a signed `FAIL` executor double, and trusted fixture runners; none is presented as live Codex evidence. Separate worker/verifier/egress Dockerfiles, a candidate-to-inspected-ID shell-free Docker driver, a fixed two-file-overlay/no-network-verifier plan, a validate-only worker entrypoint, command-backed run-capability authentication, and a Responses-only reverse broker are statically or locally tested. The driver is connected only to the prepared static lifecycle and a stateful fake daemon, not to a signed live RPC result. The supervisor seals the worker image and request maxima; memory equals memory+swap, regular-file writes and the single local log file share bounded ceilings, and one prepare/worker/verifier execution deadline is reflected in the plan. All prepared process plans explicitly use `restart=no`; the fake-daemon driver requires zero restarts and a stable ID/PID/start timestamp, with egress reobserved around worker execution and before stop. A required CPU-controller port and fake-only ledger aggregate post-baseline egress, worker, and verifier values using `bigint`, bind a static proof to the request, Docker binding, and three role identities, and delay receipt validation until proof finalization. The proof explicitly reports no cumulative enforcement, hard limit, overshoot bound, or containment, and cannot pass the live gate. The schema-v10 non-live observer contract now pins exact Docker cgroup membership to a private directory descriptor/device/inode, uses bounded no-follow reads, requires descendant quiescence and original release, and preserves cleanup-action failures; its parser and source boundary are covered offline. No real cgroup path, `cpu.stat`, baseline timing, polling latency, freeze/kill permission, or actuation has been observed, and the observer's post-start baseline cannot serve as cumulative live evidence. Cleanup has a separate bounded grace period, and controller-cleanup failure blocks the lifecycle. No immutable image has run; real restart/swap/`fsize`/log-limit behavior, the broker's internal TLS and upstream identity, process/cgroup limits, or teardown are not dynamically proved. The web process now exposes a guarded repair-run ledger and SSE timeline, but its execution port is deliberately unavailable and every current attempt records `BLOCKED / NOT_STARTED`; the host live-backend factory still rejects. The repository also executes checksum-pinned OPA 1.18.2 locally over the 41-case accepted corpus.
- A new internal Worker RPC v2 execution core binds a canonical active request to the existing orchestrator, fixed command order, complete-corpus port, review, and offline report hash. This is still contract-only: it accepts only the non-live mode but marks the structural port `UNVERIFIED_INJECTED_BACKEND`, rejects sensitive content across the completed report, is not root-exported, returns only a deeply frozen unsigned candidate with live/signing/settlement eligibility false, and is not connected to the validate-only worker entrypoint, Docker role plan, real Codex constructor, RPC response, or repair-run success transition. Its verifier callbacks are injected test ports; a production supervisor-to-verifier transport and multi-stage receipt exchange remain missing.
- The exact 41-case loop now also exists in an internal verifier-corpus candidate module, but this is not a production verifier or receipt. It consumes an injected evaluator plus caller-supplied attempt/run, command, tree, and clock observations; all are labeled unverified. It checks active v2 request continuity, canonical PolicyIR and result hashes, fixed typecheck/test order, one unchanged caller-observed tree across both commands, sensitive content, and exact corpus ordering. It is not root-exported. TypeScript AST inspection resolves supported production module references, rejects non-literal direct import/require calls and common indirect loader forms, checks package/tsconfig candidate mappings, and fails on parser errors under `src/`, `scripts/`, and `app/`; this is a repository regression guard explicitly labeled as not runtime proof or sandboxing. The candidate still cannot be admitted as policy evidence, an RPC response, a validated external run, or settlement. A verifier-owned immutable snapshot, source/build tree separation, verifier-only one-use authority and replay store, runtime entrypoint, independent review exchange, and finalized supervisor receipt remain missing.
- Worker RPC v2 CPU evidence schema v2 is an offline contract, not live CPU evidence. It strictly validates a request-bound global `CLOCK_MONOTONIC_RAW_NS` transcript, egress/worker overlap, verifier ordering, linked samples, success arithmetic, and closed pre-execution/controller/overage/containment failure outcomes. A non-exported synthetic state-machine producer derives frozen `UNSIGNED_CPU_EVIDENCE_V2_CANDIDATE` objects, rejects self-declared Linux provenance, and permanently poisons the contract session on unrecordable abort/overflow/bind-boundary errors. A separate private kernel/Docker construction now exists, but it has not produced a runtime transcript or a finalized result and is not a signing path. Synthetic unit evidence uses test signing keys; the loopback mTLS v2 path intentionally signs only typed pre-execution `FAIL`, and the generic supervisor rejects PASS. No artifact proves that Linux supplied the timestamps or samples, a controller polled or contained processes, Docker executed, cgroups were released, or cumulative CPU was enforced. Contract-implemented flags are true while observation/dynamic flags remain false. `hardLimitEnforced` and `overshootBounded` remain false, and `verify:live` remains blocked.
- Schema v15 retains the v10 observer, v11 private final-result guard, and v12 barrier/helper layers, then seals the helper artifact image ID, source hash, build-input hash, and binary hash into lifecycle v3 alongside the role images, owned networks, exact role plans, Docker receipts, helper session, system adapter, and dedicated lifecycle. A required digest-pinned compiler stage emits one root-owned `0555` AMD64 static PIE into `scratch`; direct ELF/tar checks reject an interpreter, shared-library dependency, executable stack, wrong architecture/type, oversized output, ownership drift, and mode drift. Two local WSL builds were byte-identical, but the compiler is unpinned and the report cannot promote image, install, runtime, or signing claims. The dynamic helper gate uses no pull or build network and currently fails before Docker because the immutable builder is unset. The owner snapshots the sealed binary hash and the adapter rejects any other same-FD helper client. Create ambiguity, network ownership, bind/reobservation order, quiescent teardown, and final absence rules remain as before. The available WSL cgroup root is tmpfs rather than cgroup v2, and the older static Docker gates do not invoke the private path. Artifact-image reproducibility, host installation, cross-UID bind permissions, process/FD exclusivity, Docker/cgroup identity behavior, containment, cleanup, and the validation-only role target remain dynamic gaps. No runtime transcript, finalized-evidence issuer, signer admission, PASS, or live-gate change exists.
- The v2 transport capability closes public string/copy/wrapper admission and later caller-option mutation only. Its actual TLS factory, validated scalar snapshot, defensively copied in-memory TLS material, and private `WeakSet` are co-located and no arbitrary registrar exists, but this is not a sandbox against arbitrary code already executing inside the trusted host process. Actual security still depends on TLS 1.3 peer validation and pins, the dedicated Ed25519 trust bundle, durable replay, and the disabled PASS/live gate.
- The zero-drift fixed fixture is evaluation-only. It is not post-Codex evidence.
- The current mutation score uses the reference evaluator, not OPA.
- The evidence package is intentionally `PARTIAL_OFFLINE` and `FAIL`.
- SQLite persistence, anonymous browser-session isolation, versioned Decision Queue/source writes, the six-screen web workspace, production build, and local Chrome navigation/keyboard/responsive checks exist. The separate run ledger stores only a session hash, permits 16 attempts per session, resumes bounded SSE events, and refuses to unlock a started run after an ordinary exception or transport close because those do not prove remote cleanup. Safe terminal run/event rows are pruned with 24-hour session expiry; active or poisoned rows are deliberately retained and block the single shared executor across sessions, so they require reset/operator recovery instead of silent deletion. The store is still process-local: there is no authenticated identity, distributed quota, cross-replica coordination, backup verification, cancellation/status-query protocol for recovering a lost RPC response, or operator recovery UI. Anonymous projects have a process-local 24-hour TTL and 128-project cap. The 14-to-30 text edit persists only as a blocked v5 `DRAFT`; its impact is a reference preview, not accepted PolicyIR, fresh OPA evidence, or a repair result. A full accessibility audit, multi-user authentication, container health, deployment, public URLs, video, and challenge submission remain incomplete.
- The deterministic 38-file USTAR download and 14-to-30 preview belong only to the seeded reference ambiguity choices. The archive is uncompressed and generated in memory from validated text evidence; it deliberately excludes screenshots, SQLite data, transient logs, and arbitrary files. A semantic PolicyIR fingerprint exposes a mismatch and blocks the draft when a browser accepts different valid choices; session-specific proof generation is not implemented yet.
- The evidence-download cache is process-local and holds at most one completed archive for at most 15 seconds. Every request still rereads and hashes the bounded file set before reuse; changed content, trust/freshness policy, validation failure, or live-attestation expiry prevents reuse. It reduces repeated validation/archive work but is not client throttling, shared admission control, or a multi-replica quota.
- SHA-256 evidence hashes provide integrity, not origin authentication. `LIVE_VERIFIED` additionally requires a trusted Ed25519 attestation whose private key must stay outside the repository and whose default freshness window is 24 hours. Invocation-time public keys are admitted only when their key ID and SPKI SHA-256 match `config/attestation-trust.v1.json`; that policy is deliberately unprovisioned and the live signer is not implemented yet.
- Local release receipts bind reviewed files and owner observations but are not third-party authenticity services. The strict release command reruns the offline gate, binds the submitted live URL to validated deployment/browser evidence, rejects redirects and non-public literal or DNS addresses, pins the validated address for anonymous HTTPS reachability, and checks anonymous GitHub/GitLab `HEAD` from an isolated Git configuration. It performs no publication probe when that fresh offline gate fails. Successful probes are point-in-time network observations and require approval. YouTube/Devpost availability, signed-out playback, audio clarity, rules-page content, and confirmation still require approved external browser verification immediately before publication.
- The standalone web, worker, verifier, egress, and native-helper Dockerfiles and daemon-free static contract exist, and Docker Linux server 29.1.5 is running locally with `cgroupfs` and cgroup v1. The reviewed release-host Docker CLI hash plus immutable Node and helper-builder bases are unset, so current reports stop before workload execution; this daemon would still be ineligible for worker/egress proof after identities are supplied. `helper:verify` stops before Docker until a digest-pinned compiler image is configured and already present; even a passing artifact build would not prove host installation or cgroup runtime. `container:verify` covers the web image and SQLite restart path through a canonical, contract-hashed local-Docker runner, nonce-bound image/volume/container ownership, observed IDs, pre-start restart/resource checks, and final-absence cleanup. It uses `--pull=false`, but a cold build can still require the pinned pnpm and OPA downloads and has not run against real Docker yet. `worker:verify` is the non-live worker/verifier gate with a fresh request-bound internal network, inspected-ID cleanup, exact cgroup-v2 identity, descendant-quiescence/initial-PID/release facts, and exact role-local post-exit CPU comparisons that are not aggregate enforcement. `egress:verify` is a separate non-live proxy gate with fresh internal/outbound networks, private ephemeral secrets, the same cgroup/absence proof, and a TLS 1.3 certificate probe that writes no HTTP; it also requires the sealed helper artifact identity because its lifecycle plan is v3. Both gates now require a canonical Linux cgroup-v2 supervisor before builds, but that preflight is not container isolation evidence. Proxy outbound traffic is not measured, so upstream absence is not proved. All dynamic reports currently fail before invoking Docker. The selected Docker/BuildKit and compiler identities, exact image environment/inspect compatibility, eligible cgroup-v2 host, signed real-Linux three-role cumulative CPU proof, actual egress runtime, forced termination, real Codex headers, and OpenAI upstream remain unverified.
- No project license has been selected because owner acceptance is required.
- Arbitrary repository upload/execution, multiple policy domains, multiple languages, and production auto-deployment are excluded from the MVP.
