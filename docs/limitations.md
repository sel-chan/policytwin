# Limitations

- PolicyTwin is a software verification aid, not legal advice. Human approval is required before real policy deployment.
- Only the synthetic English SaaS refund policy and bundled trusted TypeScript fixture are supported.
- The current repository has not performed a fresh GPT-5.6 request or live Codex SDK repair. The pinned SDK-compatible adapter, signed RPC client, real TLS 1.3 mutual-authentication transport, bounded supervisor, and durable SQLite replay store are covered offline with ephemeral certificates, generated test signing keys, a signed `FAIL` executor double, and trusted fixture runners; none is presented as live Codex evidence. The wire path rejects wrong or missing peer identity, unknown/noncanonical frames, declared/actual length mismatches, oversized, partial, trailing, replayed, expired, and concurrent requests, while the host retains signature, teardown, phase/corpus, immutable-command, and exact two-file tree-delta checks. Separate worker/verifier Dockerfiles and a fixed two-file-overlay/no-network-verifier run plan are statically checked, but no immutable worker/verifier image has run, the OpenAI-only egress proxy and OS executor are absent, and process limits or teardown are not dynamically proved. The web process has no live repair route, the host live-backend factory always rejects, and host command execution rejects live mode. The repository also executes checksum-pinned OPA 1.18.2 locally over the 41-case accepted corpus.
- The zero-drift fixed fixture is evaluation-only. It is not post-Codex evidence.
- The current mutation score uses the reference evaluator, not OPA.
- The evidence package is intentionally `PARTIAL_OFFLINE` and `FAIL`.
- SQLite persistence, anonymous browser-session isolation, versioned Decision Queue/source writes, the six-screen web workspace, production build, and local Chrome navigation/keyboard/responsive checks exist. Anonymous projects have a process-local 24-hour TTL and 128-project cap, but there is no authenticated identity or distributed quota. The 14-to-30 text edit persists only as a blocked v5 `DRAFT`; its impact is a reference preview, not accepted PolicyIR, fresh OPA evidence, or a repair result. A full accessibility audit, multi-user authentication, container health, deployment, public URLs, video, and challenge submission remain incomplete.
- The deterministic 38-file USTAR download and 14-to-30 preview belong only to the seeded reference ambiguity choices. The archive is uncompressed and generated in memory from validated text evidence; it deliberately excludes screenshots, SQLite data, transient logs, and arbitrary files. A semantic PolicyIR fingerprint exposes a mismatch and blocks the draft when a browser accepts different valid choices; session-specific proof generation is not implemented yet.
- SHA-256 evidence hashes provide integrity, not origin authentication. `LIVE_VERIFIED` additionally requires a trusted Ed25519 attestation whose private key must stay outside the repository and whose default freshness window is 24 hours; the live signer is not implemented yet.
- The standalone web, worker, and verifier Dockerfiles and daemon-free static contract exist, but the immutable Node base is unset and the local Docker Desktop Linux daemon is unavailable. `container:verify` covers the web image and SQLite restart path. On a Docker-eligible run, `worker:verify` fixes build-input hashes, rebuilds the current runtime plan, builds both local images without an external mutable syntax frontend, records their runtime image IDs and Docker server version, requires an internal-only worker network, reconstructs the two repaired overlays into a separate verification tree with copied paths and three content bindings, and runs fixed commands in a credential-free `network=none` verifier. Neither dynamic gate has passed; the selected Docker/BuildKit version is not yet release-verified, the egress proxy and live executor remain unimplemented, and forced termination such as `SIGKILL` cannot guarantee cleanup.
- No project license has been selected because owner acceptance is required.
- Arbitrary repository upload/execution, multiple policy domains, multiple languages, and production auto-deployment are excluded from the MVP.
