# PolicyTwin threat model

## Scope and assets

Protected assets are OpenAI/Codex credentials, repository contents, accepted policy meaning, golden cases, the canonical buggy fixture, generated evidence, command output, and submission claims.

The hosted MVP will accept policy text and structured refund cases. It will execute writes only in a fresh copy of the bundled trusted fixture. Arbitrary repositories, arbitrary shell commands, and production auto-deployment are outside scope.

## Trust boundaries

1. Browser to server: policy text, ambiguity choices, and case inputs are untrusted.
2. Server to model: prompts include untrusted policy prose under a strict schema contract.
3. Server to OPA: only deterministically compiled Rego and strictly validated integer/boolean input may cross.
4. Server to repair supervisor: a concrete TLS 1.3 transport mutually authenticates CA-chained, name-checked, certificate-pinned peers with fixed ALPN. It carries one canonical length-framed RPC containing only validated policy evidence, opaque request identity, immutable digests/manifests, closed IDs, and resource policy. SQLite rejects request-ID or nonce reuse across supervisor restarts.
5. Worker to egress broker: the worker receives only a short-lived, run-bound capability through command-backed Codex authentication. The broker alone receives the provider credential through an external read-only file mount and implements a fixed Responses API reverse route, not generic `CONNECT` forwarding.
6. Worker supervisor to child process: repair and immutable verification use separate workspaces; model credentials are removed from fixture processes, output is bounded, and teardown must be supervisor-observed before signing.
7. Evidence to UI/submission: every claim must map to a hash-covered artifact with explicit provenance; `LIVE_VERIFIED` additionally requires a trusted Ed25519 attestation bound to the evidence hash, run ID, and timestamp.
8. Linux CPU evidence to Worker RPC v2: only the separate v2 protocol/signature/mTLS profile, durable replay store, and structured live-purpose key registry with material distinct from v1 may carry future live evidence. CPU evidence schema v2 derives Docker and partial-attempt bindings from the request and observed role identities, signs one globally ordered monotonic event transcript, and closes pre-execution, non-CPU, controller, over-budget-contained, and containment-incomplete outcomes. An internal, non-exported synthetic producer exercises this contract but emits only frozen unsigned/non-live wrappers with current signing eligibility false and rejects self-declared Linux provenance. The enclosed raw evidence remains parser-valid contract data, not provenance or signer authorization. The current generic supervisor signs FAIL only; a capability-bound private Linux construction exists in source but has no runtime-verified or finalized result. V1, static fake, nullable, unsigned, replayed, or downgraded values are never upgraded.
9. Host to Worker RPC v2 transport: the public authentication-mode string is diagnostic only. The concrete v2 mTLS client module owns a private `WeakSet`, snapshots validated scalar inputs plus defensive copies of in-memory CA/certificate/key material, and only its actual factory freezes and adds an admissible object; no arbitrary registrar exists. V1 objects, self-declarations, copies, wrappers, and later caller option mutations cannot alter the admitted connection profile before request construction.
10. Private Linux adapter to signer: schema v15 requires one deeply frozen lifecycle-v3 plan that seals the helper artifact image/source/build/binary identities in addition to the role images, private networks, role plans, Docker-owner, reobservation/removal receipts, helper session, system adapter, and dedicated lifecycle. The owner snapshots the sealed helper binary hash and the adapter rejects a mismatched same-FD helper client. The role/host barrier capability, local-build report, artifact image alone, raw helper response, synthetic candidate, non-live observer, adapter diagnostic, copied plan, wrapper, or JSON round trip never becomes signing authority. The finalized-evidence identity set still has no issuer; any future signer must require the exact private object issued only after dynamically verified Docker/cgroup cleanup, owned-network absence, and controller stop.
11. Browser session to repair-run ledger: the run database stores only a SHA-256 session binding, never the cookie. Creation is same-origin/CSRF protected, idempotent by a canonical client UUID, serialized with policy mutations, and reconstructs the exact trusted input after the gate is held. Reads and SSE require the same session binding; event cursors are monotonic and bounded. The exact local `rr_<32 hex>` suffix becomes the signed v2 `requestId`; the branded client consumes the authenticated result before issuing one fresh private settlement, and the ledger stores the request ID, execution binding, and completion time. A shaped, copied, pre-issued, replayed, cross-run, mismatched-input, partial-phase, wrong-file/command/corpus, or unapproved result cannot advance the run.

## Threats and controls

| Threat | Current control | Remaining work |
|---|---|---|
| Prompt injection in policy text | Interpreter prompts treat policy as data; outputs use a closed IR and strict validation | Verify current Responses API Structured Outputs live |
| Cross-site, replayed, or storage-exhausting workspace access | Seeded policy scope, anonymous hashed project IDs, same-origin-only session creation, exact configured production origin, HttpOnly SameSite session/CSRF cookies, matching custom header, strict version paths/bodies, ten-second body limits, optimistic concurrency, replay-safe identical writes, 24-hour cleanup, and a 128-project process cap | Add authenticated identity, shared quotas/rate limiting, and coordinated cleanup before multi-instance hosting |
| Forged, replayed, cross-session, overlapping, or storage-exhausting repair runs | Separate session-hashed SQLite ledger; exact current v4 reference input reconstruction inside the shared mutation gate; canonical UUID idempotency; per-session 16-attempt history; one global active/fail-stop executor latch; session-bound GET/SSE with `Last-Event-ID`; closed event phases; injective local-run-to-signed-request binding; exact branded client and one-use settlement identities; persisted request/binding/completion provenance; exact two-file/two-command/full-corpus/review checks. Expiry prunes safe terminal runs and cascading events, but retains active/poisoned rows globally. Running state remains `LIVE_EXECUTION_UNVERIFIED`; timeout enters cleanup-pending, and any transport failure or settlement without run-bound authenticated cleanup remains persistent `POISONED` fail-stop admission | Wire the unavailable execution port only after immutable Linux runtime evidence and the still-disabled PASS/finalization boundary are admitted; add authenticated distributed quotas and operator recovery for a deployed multi-instance service |
| Static reference evidence shown for different accepted choices | Proof compares a deterministic PolicyIR meaning fingerprint; Change Impact blocks v5 when the current validated meaning differs from the seeded reference | Generate and attest session-specific evidence after live interpretation/verification exists |
| Model emits executable code | `PolicyIR` has no code field; deterministic compiler owns Rego | Live schema conformance evidence |
| Path traversal or canonical fixture modification | Relative and absolute SDK event paths normalize only within the managed root; RPC frames reject host paths; repair copies live under one checked root; every phase snapshots bounded UTF-8 files and directories; read-only changes fail; canonical hash is rechecked. Before any run file is created, the worker verifier rejects a linked/non-directory repository root, `.tmp`, or `worker-runs` parent and then rechecks the physical run directory as a direct managed child. The launch builder rejects traversal, symlinked roots, incomplete layouts, mutable images, and mounts outside the exact baseline/overlay contract. The Docker observer compares every bind source/destination/write flag against that admitted plan | Run the prepared images under Docker, validate the actual daemon path mapping and namespace/cgroup state, reject links/devices again immediately before create, and verify signed teardown receipts |
| Arbitrary command execution or cross-run Docker resource confusion | SDK phases reject command events; the public host live factory still rejects. The older static Docker runner and the separate private v15 runner both pin a canonical absolute CLI and platform-local daemon, use argv with no shell, and apply exact environments plus bounded output/deadlines. The private path accepts only a deeply frozen factory-issued lifecycle plan, creates and independently inspects its own two networks, and derives exact role plans internally. The owner preflights unique names, marks side effects unresolved before create, promotes a returned ID only after ownership inspection, permits exact-name recovery only as cleanup authority, reobserves identity around helper binding, serializes start/issue/remove operations, and requires container plus network ID/name absence before cleanup completes | Run both dynamic gates and the private construction on the selected Linux supervisor, inject daemon/create timeout and restart races, verify daemon/BuildKit and cgroup identities plus real limit violations, and connect the trusted minimal configuration provider to mTLS only after signed live evidence is possible |
| Forged, replaced, or descendant-incomplete cgroup observation | The schema-v10 non-live observer admits only final Docker-owned cgroup components, verifies cgroup v2, pins one `O_DIRECTORY | O_NOFOLLOW` descriptor plus path/device/inode identity in a private `WeakMap`, reads only allowlisted files through that descriptor with an actual byte cap, and rejects forged/finalized handles. Schema v15 separately seals the helper artifact binary hash, admits only a matching same-FD client, binds only an owner-issued one-shot Docker reobservation, rechecks pidfd/cgroup identity before every actuation, enforces one global RAW clock, requires quiescence before release, and sandwiches bind with Docker observations. Full uint64 `usage_usec` arithmetic rejects regression | Build and pin the artifact image, then run the private construction on the selected Linux daemon and validate admitted Docker path forms, descriptor/pidfd semantics, identity drift, descendant survival, containment, and cleanup. Neither the older observer, local build, artifact alone, nor unexecuted v15 source is cumulative evidence or signer authorization |
| SSRF, proxy abuse, or provider-key theft | The broker rejects `CONNECT`, absolute/query targets, arbitrary hosts, unknown or duplicate headers, transfer framing, oversized payloads, redirects, and compression. It admits only bounded JSON `POST /v1/responses`; the upstream hostname is constant, every A result must be public, the chosen IP is pinned, and TLS still checks `api.openai.com`. The worker has only a lease-limited capability and the provider credential/TLS key must be mounted from outside the repository. Prepared containers explicitly use `restart=no`; the supervisor requires zero restarts, pins the egress ID/PID/start timestamp, and rejects the worker result if that running instance changes. A separate dynamic gate validates only the internal TLS 1.3 CA/SAN/fingerprint path; its probe writes no HTTP, while proxy outbound traffic remains `NOT_MEASURED` | Run that TLS gate on immutable images, confirm the restart identity checks on the real daemon, then separately observe approved live Codex headers, OpenAI DNS/SNI/certificate identity, lease consumption, and upstream traffic; add release-platform egress/firewall controls because the destination restriction is application-layer |
| Credential or sensitive-data leakage into a model, fixture, command receipt, or proof | Every prompt context and canonical NUL-free UTF-8 fixture file is scanned before an SDK turn; the prepared SDK environment contains only a proxy-token helper and CA path, never a provider key; verification commands use a separate credential-free environment; command output, model fields, diffs, and archives reject or redact credential-shaped and personal-path content. RPC credential scans cover parsed semantic `{model,input}` and `report/error` fields before transport or trust, while complete envelopes retain canonical UTF-8, NUL, size, closed-shape, host-path, binding, and signature checks. Opaque nonces and signatures use format/cryptographic validation instead of token-pattern scans, avoiding random false rejection without exposing semantic policy text | Re-verify the same controls inside running containers and release-platform logs; include every future semantic RPC field in the scanned projection |
| Failed repair stream leaves unreviewed writes | Any repair error poisons the disposable workspace, clears the review receipt, snapshots the observable delta when safe, and blocks every later phase on that backend | External worker must terminate the entire process tree and delete the mounted workspace on every exit path |
| Test weakening or evidence edits | Actual filesystem deltas define changed files; the accepted corpus must equal the exact server-owned 41-case digest; the exact two-file repair set cannot be model-expanded or narrowed; the test file must match the digest of the server-owned D01-D03 assertions with all three skips removed; model-reported regression links are excluded; test execution must preserve the typechecked tree's content, structure, modes, and mtimes; each command-passing attempt receives a PolicyIR/corpus/tree-bound receipt and only final 41/41 reaches review; live evidence cross-checks the complete command history, generated `dist` allowlist, prompt/schema hashes, final receipt, and a canonical diff reconstructed byte-for-byte from the attested before/after contents | The same immutable runner must execute inside the external OS sandbox for live work because arbitrary model-authored tests and host execution are not proof |
| Attacker rewrites all evidence and recomputes hashes | Validator derives Rego, OPA agreement, differential counts, mutation score, and traceability from source artifacts; `LIVE_VERIFIED` requires a trusted detached Ed25519 signature | Provision and rotate the live signing key outside Git/hosting logs |
| Secret or transient file enters a downloadable archive | Archive input is the exact 38-file required allowlist, never a directory listing; every request rereads bounded UTF-8 inputs and binds every byte plus trust/freshness policy into the cache key; semantic validation, attestation checks, byte caps, credential/private-key/bearer/personal-path rejection, and deterministic USTAR generation run before a cache miss can respond | Re-run the same guard against fresh live receipts and release-platform logs |
| Recorded evidence shown as fresh | Mandatory execution modes; `PARTIAL_OFFLINE` package is `FAIL`; post-repair drift remains null | UI labels and freshness checks |
| Denial of service | Per-command 30-second timeout and 2 MiB process buffer; at most two repair attempts; each SDK phase has abort and byte/event limits; the TLS supervisor bounds frames/connections and one active repair. The web ledger permits one active/fail-stop run, 16 attempts, and 64 events per session. Its wall timeout is followed by bounded settlement; an executor that ignores cancellation poisons admission instead of allowing overlap. Evidence downloads have fixed file/aggregate bounds, one active archive build, and at most one validated completed archive reused for 15 seconds; changed keys, failed builds, invalid metadata, size violations, or live-attestation expiry cannot enter or extend that cache. The egress lease lasts at most 15 minutes and 64 requests, with 1 MiB request and 8 MiB response caps. The supervisor seals the worker image and resource maxima. Worker memory+swap, PID, regular-file size, one-file local-log size, output, and one prepare/worker/verifier execution deadline bind to the admitted request; cleanup has a separate bounded grace period and auxiliary verifier/egress profiles stay separate. Docker CPU shares are only a rate defense. The v14 private construction includes serial native sampling, freeze/kill/quiescence, sticky cleanup, independently bounded ambiguous-create recovery, owned-network teardown, and bounded emergency termination, but it has not run; the fake ledger still reports no cumulative enforcement, hard limit, overshoot bound, or containment | Prove swap, `fsize`, log rotation, create-timeout recovery, network cleanup, and cleanup under real limit violations; dynamically validate the private controller's real `cpu.stat` samples, actuation, containment, and teardown on Linux, and admit only a signed structured proof before live wiring; also add network-edge connection/rate limits, SDK stderr/single-JSONL-line hard limits, shared admission control, and daemon-backed forced-termination evidence |
| CPU-evidence substitution, downgrade, or replay | Worker RPC v2 has a separate signature domain, client-derived execution binding, request/role-derived Docker or attempt binding, structured live-purpose trust, distinct v1/v2 key fingerprints, mutual TLS-only transport, v2 ALPN/frame magics, durable SQLite request replay, exact-key receipts, required CPU evidence v2 for PASS and FAIL, and a fail-only generic supervisor. The parser rejects v1/static/nullable evidence, foreign fields, noncanonical bindings, non-monotonic or unlinked events/samples, impossible role/controller lifecycles, phase/code mismatch, aggregate/budget drift, incomplete cleanup promoted to success, and hard-limit/overshoot promotion. The internal synthetic producer is not root-exported, rejects Linux provenance, and marks every wrapper signing-ineligible; raw parser-valid evidence is explicitly not authorization. The private runtime construction is also not signer authority and has no final-result issuer | Dynamically validate the exact private construction under real under-budget, over-budget, identity-drift, controller, containment, and cleanup failures; add a separate private final-result issuer only after those gates pass, and enable PASS only after the live gate validates those fresh receipts |
| CPU adapter or finalized-evidence capability forgery | Schema v15 keeps compile-time brands separate from runtime authority. Private identity sets admit only the exact recursively frozen lifecycle-v3 plan, real Docker CLI runner/owner, owner-observed networks, internally derived role plans, one-shot Docker reobservations, a helper client matching the sealed artifact hash, bound helper roles, the system adapter, and the private lifecycle; a separate finalized-evidence set still has no issuer. Docker identity is reobserved around helper binding, actuation revalidates pidfd/cgroup identity, removal requires independently verified ownership and container/network name plus ID absence, and forced helper termination requires an opaque all-Docker-roles-absent receipt. Plain objects, local-build reports, artifact images by themselves, copied plans, wrappers, JSON round trips, harness results, raw helper frames, synthetic candidates, raw evidence, and non-live observations remain ineligible | Build and pin the helper artifact image, verify cross-UID mount permissions and that only the trusted launcher exists with no surviving writable receipt FD before release, dynamically validate the full lifecycle and network ownership on cgroup v2, then add a private final-result issuer and require its exact identity at the still-disabled signer |
| Transport self-declaration, copying, wrapping, or option mutation | The v2 client checks object identity in a module-private `WeakSet`; only the co-located concrete factory validates and privately snapshots scalar and in-memory TLS inputs, freezes and adds the exact object, and no arbitrary registrar exists. V1 factory results, fake objects, copies, wrappers, root-export access, transport mutation, scalar redirection, and Buffer/array corruption are regression tested through real TLS factory transports | Keep the factory/assertion module outside the package barrel and preserve defensive input copies; replace with a native/private runtime handle if the trusted host-process boundary changes |
| Supply-chain compromise | Exact dependency lock, production audit, checksum-pinned OPA, split web/worker/verifier/egress Dockerfiles with no mutable base-image or external mutable syntax-frontend fallback, hashes over every copied role input, and bare runtime SHA-256 image IDs | Verify the immutable Node base and selected Docker/BuildKit version or a digest-pinned frontend, record runtime image IDs, then rerun release-platform license/audit/image checks |
| Secret in Git history | Current/history scanner reports only file/commit context and uses test sentinels | Run again before publication and after live artifacts |

## Security invariants

- No model or user text selects a filesystem path or executable directly.
- `expected-fixed` is evaluation-only and never enters repair context.
- The canonical buggy fixture remains reproducible.
- `ALLOW`, `DENY`, and `REVIEW` are the only decisions.
- A missing external gate cannot be converted to `PASS` by documentation.
- A Docker name or matching label alone is never ownership. Normal execution authority requires the ID returned by this run's successful create and an independent exact name/label inspection. If create side effects are ambiguous, an exact-name lookup may admit exactly one independently name/label-verified ID for cleanup only; it grants no start or execution authority, and zero/foreign/ambiguous or unobservable results remain a sticky cleanup failure until absence is proved.
- A worker receipt is untrusted unless the exact owned egress container retains restart policy `no`, restart count zero, and the pinned running PID/start timestamp through the worker execution boundary. Worker, verifier, and egress logs are untrusted until the same instance is observed stopped with PID zero before and after log collection.
- A self-generated SHA-256 manifest cannot authenticate a live run; the private attestation key remains outside the repository, and an invocation-time public key is trusted only when its key ID and SPKI SHA-256 match the reviewed release trust file.
- Parser-accepted or test-signed Worker RPC v2 CPU evidence is not runtime evidence. Only a fresh receipt from an independently admitted live-purpose key and real Linux controller may be considered, and hard-limit or bounded-overshoot claims remain false.
- Logs, screenshots, and proof packages must not contain credentials or personal absolute paths.

## Residual risk

This is not a completed release security review. SQLite/browser/OPA/build/archive/E2E checks, fake-stream SDK controls, signed RPC bindings, real loopback mTLS framing/cancellation, durable replay rejection, static split-container contracts, local fake-upstream broker requests, and a stateful fake-daemon Docker lifecycle are covered. The host live factory still rejects; no actual SDK is constructed by the RPC path and no web route invokes repair. Docker Linux server 29.1.5 is running locally with `cgroupfs` and cgroup v1, but no worker/verifier/egress image has run because the immutable Node and helper-builder images are unset. The worker and TLS-only reports therefore remain `FAIL` with `dockerInvoked:false`; after image pinning, the early Linux/cgroup-v2 supervisor preflight would also reject this host before builds. That preflight is eligibility only, not container isolation evidence. The TLS-only report does not measure proxy outbound traffic. Production identity/storage/quotas, real Docker mounts/namespaces/cgroups/process trees, actual internal TLS, real OpenAI DNS/TLS/upstream behavior, signing-key custody, hosting, and deployment secrets remain untested. `artifacts/evidence/security-review.md` therefore stays `NOT_RUN`, and the partial proof package remains `FAIL`.
