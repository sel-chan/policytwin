# PolicyTwin threat model

## Scope and assets

Protected assets are OpenAI/Codex credentials, repository contents, accepted policy meaning, golden cases, the canonical buggy fixture, generated evidence, command output, and submission claims.

The hosted MVP will accept policy text and structured refund cases. It will execute writes only in a fresh copy of the bundled trusted fixture. Arbitrary repositories, arbitrary shell commands, and production auto-deployment are outside scope.

## Trust boundaries

1. Browser to server: policy text, ambiguity choices, and case inputs are untrusted.
2. Server to model: prompts include untrusted policy prose under a strict schema contract.
3. Server to OPA: only deterministically compiled Rego and strictly validated integer/boolean input may cross.
4. Server to repair supervisor: a concrete TLS 1.3 transport mutually authenticates CA-chained, name-checked, certificate-pinned peers with fixed ALPN. It carries one canonical length-framed RPC containing only validated policy evidence, opaque request identity, immutable digests/manifests, closed IDs, and resource policy. SQLite rejects request-ID or nonce reuse across supervisor restarts.
5. Worker to egress broker: the worker receives only a short-lived, run-bound capability through command-backed Codex authentication. The broker alone receives the provider credential through an external read-only file mount and implements a fixed Responses API reverse route, not generic `CONNECT` forwarding.
6. Worker supervisor to child process: repair and immutable verification use separate workspaces; model credentials are removed from fixture processes, output is bounded, and teardown must be supervisor-observed before signing.
7. Evidence to UI/submission: every claim must map to a hash-covered artifact with explicit provenance; `LIVE_VERIFIED` additionally requires a trusted Ed25519 attestation bound to the evidence hash, run ID, and timestamp.
8. Linux CPU evidence to Worker RPC v2: only the separate v2 protocol/signature/mTLS profile, durable replay store, and structured live-purpose key registry with material distinct from v1 may carry future live evidence. CPU evidence schema v2 derives Docker and partial-attempt bindings from the request and observed role identities, signs one globally ordered monotonic event transcript, and closes pre-execution, non-CPU, controller, over-budget-contained, and containment-incomplete outcomes. The current generic supervisor signs FAIL only, and no dedicated Linux producer exists. V1, static fake, nullable, unsigned, replayed, or downgraded values are never upgraded.
9. Host to Worker RPC v2 transport: the public authentication-mode string is diagnostic only. The concrete v2 mTLS client module owns a private `WeakSet`, snapshots validated scalar inputs plus defensive copies of in-memory CA/certificate/key material, and only its actual factory freezes and adds an admissible object; no arbitrary registrar exists. V1 objects, self-declarations, copies, wrappers, and later caller option mutations cannot alter the admitted connection profile before request construction.

## Threats and controls

| Threat | Current control | Remaining work |
|---|---|---|
| Prompt injection in policy text | Interpreter prompts treat policy as data; outputs use a closed IR and strict validation | Verify current Responses API Structured Outputs live |
| Cross-site, replayed, or storage-exhausting workspace access | Seeded policy scope, anonymous hashed project IDs, same-origin-only session creation, exact configured production origin, HttpOnly SameSite session/CSRF cookies, matching custom header, strict version paths/bodies, ten-second body limits, optimistic concurrency, replay-safe identical writes, 24-hour cleanup, and a 128-project process cap | Add authenticated identity, shared quotas/rate limiting, and coordinated cleanup before multi-instance hosting |
| Static reference evidence shown for different accepted choices | Proof compares a deterministic PolicyIR meaning fingerprint; Change Impact blocks v5 when the current validated meaning differs from the seeded reference | Generate and attest session-specific evidence after live interpretation/verification exists |
| Model emits executable code | `PolicyIR` has no code field; deterministic compiler owns Rego | Live schema conformance evidence |
| Path traversal or canonical fixture modification | Relative and absolute SDK event paths normalize only within the managed root; RPC frames reject host paths; repair copies live under one checked root; every phase snapshots bounded UTF-8 files and directories; read-only changes fail; canonical hash is rechecked. Before any run file is created, the worker verifier rejects a linked/non-directory repository root, `.tmp`, or `worker-runs` parent and then rechecks the physical run directory as a direct managed child. The launch builder rejects traversal, symlinked roots, incomplete layouts, mutable images, and mounts outside the exact baseline/overlay contract. The Docker observer compares every bind source/destination/write flag against that admitted plan | Run the prepared images under Docker, validate the actual daemon path mapping and namespace/cgroup state, reject links/devices again immediately before create, and verify signed teardown receipts |
| Arbitrary command execution or cross-run Docker resource confusion | SDK phases reject command events; host live construction rejects. The Docker runner pins a canonical absolute CLI and the platform-local daemon, uses argv with no shell, an exact environment, bounded output/deadlines, and a closed verb/weakening denylist. Every resource name and PolicyTwin label is request/nonce-bound, existing names are never adopted, and create stdout becomes cleanup authority only after independent ID/name/label inspection. Supervisor parsing closes the sealed worker image and maximum limits, users, entrypoints, working directories, environments, namespaces, devices, security options, memory+swap, file/log ceilings, bind propagation, tmpfs, ports, exact network IDs/membership plus required aliases, and teardown; cleanup requires both ID-filter and binding/role absence | Run both dynamic gates on the selected Linux supervisor, verify daemon/BuildKit and cgroup identities plus real limit violations, expand timeout/abort observations, and connect the trusted minimal configuration provider to mTLS only after signed live evidence is possible |
| SSRF, proxy abuse, or provider-key theft | The broker rejects `CONNECT`, absolute/query targets, arbitrary hosts, unknown or duplicate headers, transfer framing, oversized payloads, redirects, and compression. It admits only bounded JSON `POST /v1/responses`; the upstream hostname is constant, every A result must be public, the chosen IP is pinned, and TLS still checks `api.openai.com`. The worker has only a lease-limited capability and the provider credential/TLS key must be mounted from outside the repository. Prepared containers explicitly use `restart=no`; the supervisor requires zero restarts, pins the egress ID/PID/start timestamp, and rejects the worker result if that running instance changes. A separate dynamic gate validates only the internal TLS 1.3 CA/SAN/fingerprint path; its probe writes no HTTP, while proxy outbound traffic remains `NOT_MEASURED` | Run that TLS gate on immutable images, confirm the restart identity checks on the real daemon, then separately observe approved live Codex headers, OpenAI DNS/SNI/certificate identity, lease consumption, and upstream traffic; add release-platform egress/firewall controls because the destination restriction is application-layer |
| Credential or sensitive-data leakage into a model, fixture, command receipt, or proof | Every prompt context and canonical NUL-free UTF-8 fixture file is scanned before an SDK turn; the prepared SDK environment contains only a proxy-token helper and CA path, never a provider key; verification commands use a separate credential-free environment; command output, model fields, diffs, and archives reject or redact credential-shaped and personal-path content | Re-verify the same controls inside running containers and release-platform logs |
| Failed repair stream leaves unreviewed writes | Any repair error poisons the disposable workspace, clears the review receipt, snapshots the observable delta when safe, and blocks every later phase on that backend | External worker must terminate the entire process tree and delete the mounted workspace on every exit path |
| Test weakening or evidence edits | Actual filesystem deltas define changed files; the accepted corpus must equal the exact server-owned 41-case digest; the exact two-file repair set cannot be model-expanded or narrowed; the test file must match the digest of the server-owned D01-D03 assertions with all three skips removed; model-reported regression links are excluded; test execution must preserve the typechecked tree's content, structure, modes, and mtimes; each command-passing attempt receives a PolicyIR/corpus/tree-bound receipt and only final 41/41 reaches review; live evidence cross-checks the complete command history, generated `dist` allowlist, prompt/schema hashes, final receipt, and a canonical diff reconstructed byte-for-byte from the attested before/after contents | The same immutable runner must execute inside the external OS sandbox for live work because arbitrary model-authored tests and host execution are not proof |
| Attacker rewrites all evidence and recomputes hashes | Validator derives Rego, OPA agreement, differential counts, mutation score, and traceability from source artifacts; `LIVE_VERIFIED` requires a trusted detached Ed25519 signature | Provision and rotate the live signing key outside Git/hosting logs |
| Secret or transient file enters a downloadable archive | Archive input is the exact 38-file required allowlist, never a directory listing; semantic validation, attestation checks, byte caps, credential/private-key/bearer/personal-path rejection, and deterministic USTAR generation run before response | Re-run the same guard against fresh live receipts and release-platform logs |
| Recorded evidence shown as fresh | Mandatory execution modes; `PARTIAL_OFFLINE` package is `FAIL`; post-repair drift remains null | UI labels and freshness checks |
| Denial of service | Per-command 30-second timeout and 2 MiB process buffer; at most two repair attempts; each SDK phase has abort and byte/event limits; the TLS supervisor bounds frames/connections and one active repair. The egress lease lasts at most 15 minutes and 64 requests, with 1 MiB request and 8 MiB response caps. The supervisor seals the worker image and resource maxima. Worker memory+swap, PID, regular-file size, one-file local-log size, output, and one prepare/worker/verifier execution deadline bind to the admitted request; cleanup has a separate bounded grace period and auxiliary verifier/egress profiles stay separate. Docker CPU shares are only a rate defense. A required controller port and fake-only `bigint` ledger prove static three-role aggregation/order and receipt blocking, but explicitly report no cumulative enforcement, hard limit, overshoot bound, or containment | Prove swap, `fsize`, log rotation, and cleanup under real limit violations; implement a supervisor-owned cgroup controller that binds real `cpu.stat` samples and actuation to the request/container identities, validate containment and teardown on Linux, and admit only a signed structured proof before live wiring; also add network-edge connection/rate limits, SDK stderr/single-JSONL-line hard limits, shared admission control, and daemon-backed forced-termination evidence |
| CPU-evidence substitution, downgrade, or replay | Worker RPC v2 has a separate signature domain, client-derived execution binding, request/role-derived Docker or attempt binding, structured live-purpose trust, distinct v1/v2 key fingerprints, mutual TLS-only transport, v2 ALPN/frame magics, durable SQLite request replay, exact-key receipts, required CPU evidence v2 for PASS and FAIL, and a fail-only generic supervisor. The parser rejects v1/static/nullable evidence, foreign fields, noncanonical bindings, non-monotonic or unlinked events/samples, impossible role/controller lifecycles, phase/code mismatch, aggregate/budget drift, incomplete cleanup promoted to success, and hard-limit/overshoot promotion | Implement the dedicated Linux producer, dynamically observe the signed global transcript and closed outcomes under real under-budget, over-budget, identity-drift, controller, containment, and cleanup failures, then enable PASS only after the live gate validates those fresh receipts |
| Transport self-declaration, copying, wrapping, or option mutation | The v2 client checks object identity in a module-private `WeakSet`; only the co-located concrete factory validates and privately snapshots scalar and in-memory TLS inputs, freezes and adds the exact object, and no arbitrary registrar exists. V1 factory results, fake objects, copies, wrappers, root-export access, transport mutation, scalar redirection, and Buffer/array corruption are regression tested through real TLS factory transports | Keep the factory/assertion module outside the package barrel and preserve defensive input copies; replace with a native/private runtime handle if the trusted host-process boundary changes |
| Supply-chain compromise | Exact dependency lock, production audit, checksum-pinned OPA, split web/worker/verifier/egress Dockerfiles with no mutable base-image or external mutable syntax-frontend fallback, hashes over every copied role input, and bare runtime SHA-256 image IDs | Verify the immutable Node base and selected Docker/BuildKit version or a digest-pinned frontend, record runtime image IDs, then rerun release-platform license/audit/image checks |
| Secret in Git history | Current/history scanner reports only file/commit context and uses test sentinels | Run again before publication and after live artifacts |

## Security invariants

- No model or user text selects a filesystem path or executable directly.
- `expected-fixed` is evaluation-only and never enters repair context.
- The canonical buggy fixture remains reproducible.
- `ALLOW`, `DENY`, and `REVIEW` are the only decisions.
- A missing external gate cannot be converted to `PASS` by documentation.
- A Docker name or matching label is never ownership; only an ID returned by this run's successful create and then independently matched to the expected name and labels authorizes later mutation or cleanup.
- A worker receipt is untrusted unless the exact owned egress container retains restart policy `no`, restart count zero, and the pinned running PID/start timestamp through the worker execution boundary. Worker, verifier, and egress logs are untrusted until the same instance is observed stopped with PID zero before and after log collection.
- A self-generated SHA-256 manifest cannot authenticate a live run; the private attestation key remains outside the repository.
- Parser-accepted or test-signed Worker RPC v2 CPU evidence is not runtime evidence. Only a fresh receipt from an independently admitted live-purpose key and real Linux controller may be considered, and hard-limit or bounded-overshoot claims remain false.
- Logs, screenshots, and proof packages must not contain credentials or personal absolute paths.

## Residual risk

This is not a completed release security review. SQLite/browser/OPA/build/archive/E2E checks, fake-stream SDK controls, signed RPC bindings, real loopback mTLS framing/cancellation, durable replay rejection, static split-container contracts, local fake-upstream broker requests, and a stateful fake-daemon Docker lifecycle are covered. The host live factory still rejects; no actual SDK is constructed by the RPC path and no web route invokes repair. No worker/verifier/egress image has run because immutable images and the Docker daemon are unavailable. The worker and TLS-only reports therefore remain `FAIL` with `dockerInvoked:false`; the latter does not measure proxy outbound traffic. Production identity/storage/quotas, real Docker mounts/namespaces/cgroups/process trees, actual internal TLS, real OpenAI DNS/TLS/upstream behavior, signing-key custody, hosting, and deployment secrets remain untested. `artifacts/evidence/security-review.md` therefore stays `NOT_RUN`, and the partial proof package remains `FAIL`.
